## 들어가며

Claude Code 생태계에는 "코드 품질을 높이자"는 같은 목적을 가진 도구가 여러 개 있다.

- Karpathy Guidelines — 행동 원칙
- Superpowers — 개발 방법론 프레임워크
- code-simplifier — 자동 리팩토링 에이전트
- feature-dev — 기능 개발 워크플로우

이름만 보면 비슷하지만 실제 역할은 다르다. 각각이 무엇이고 어떻게 다른지, 언제 써야 하는지 정리했다.


## 먼저 용어부터: 스킬 vs 플러그인 vs 에이전트

Claude Code에서 자주 혼용되는 용어를 먼저 정리하자.

| 개념 | 설명 | 호출 방법 |
|------|------|-----------|
| 스킬 (Skill) | 프롬프트 파일. Claude의 행동을 제어하는 지침서 | `/skill-name`으로 호출 |
| 플러그인 (Plugin) | 스킬 + 훅 + MCP 서버 등을 묶은 패키지 | `claude plugin add`로 설치 |
| 에이전트 (Agent) | Task 도구로 실행되는 서브프로세스. 독립 맥락에서 작업 | 스킬/플러그인 내부에서 자동 실행 |

스킬은 플러그인의 부분집합이다. 모든 플러그인은 스킬을 포함할 수 있고, 스킬은 단독으로도 존재할 수 있다.


## 4종 도구의 정체

### 1. Karpathy Guidelines — 스킬 (단독)

```
유형: 스킬 (단일 마크다운 파일)
설치: ~/.claude/skills/ 또는 CLAUDE.md에 직접 작성
호출: /karpathy-guidelines
출처: 커뮤니티 (Andrej Karpathy의 관찰에서 파생)
```

4가지 행동 원칙을 프롬프트로 주입한다.

1. Think Before Coding — 가정하지 말고, 트레이드오프를 드러내라
2. Simplicity First — 최소 코드, 추측성 기능 금지
3. Surgical Changes — 건드려야 할 것만 건드려라
4. Goal-Driven Execution — 검증 가능한 성공 기준을 정의하라

"~하지 마라"는 제약을 거는 방식이다. LLM의 나쁜 습관을 사전에 차단한다.


### 2. Superpowers — 플러그인 (14개 스킬 모듈)

```
유형: 플러그인 (스킬 14개 + 훅 + 부트스트랩)
설치: claude plugin add superpowers
호출: 자동 활성화 (1% 룰)
출처: Jesse Vincent (obra) → 2026-01 Anthropic 공식 마켓플레이스 등재
```

14개 스킬이 세션 시작 시 자동으로 주입되고, 상황에 따라 활성화된다.

| 카테고리 | 주요 스킬 |
|----------|-----------|
| 설계 | brainstorming, writing-plans |
| 구현 | test-driven-development, executing-plans |
| 품질 | requesting-code-review, verification-before-completion |
| 디버깅 | systematic-debugging |
| 협업 | subagent-driven-development, dispatching-parallel-agents |
| Git | using-git-worktrees, finishing-a-development-branch |

"이렇게 일하라"는 프로세스를 강제한다. TDD, 체계적 디버깅, 서브에이전트 리뷰 같은 것들.

특이한 건 1% 룰이다.

> "스킬이 적용될 가능성이 1%라도 있으면, 반드시 호출해야 한다."

선택이 아니라 의무로 다룬다. `Iron Law`, `HARD-GATE` 같은 표현으로 LLM의 합리화를 차단한다.


### 3. code-simplifier — 플러그인 (자율 에이전트)

```
유형: 플러그인 (에이전트 1개)
설치: claude plugin add code-simplifier
호출: 자동 실행 (코드 수정 감지 시)
출처: Anthropic 공식 플러그인
```

코드가 수정되면 자동으로 정리한다.

- 중첩 깊이 줄이기
- 중복 코드 제거
- 변수/함수 네이밍 개선
- 중첩 삼항연산자 → switch/if-else 변환
- ES modules, React 베스트 프랙티스 적용

코드 작성 후에 도는 리팩토링이라 보면 된다.


### 4. feature-dev — 스킬 (에이전트 3종 포함)

```
유형: 스킬 (7단계 워크플로우 + 3종 에이전트)
설치: Claude Code 내장 (또는 커뮤니티 스킬)
호출: /feature-dev
출처: 커뮤니티
```

새 기능 개발을 7단계로 구조화하고, 단계별로 전문 에이전트를 투입한다.

```
Phase 1: Discovery           → 요구사항 파악
Phase 2: Codebase Exploration → code-explorer 에이전트로 코드 분석
Phase 3: Clarifying Questions → 모호한 점 질문
Phase 4: Architecture Design  → code-architect 에이전트로 설계
Phase 5: Implementation       → 구현
Phase 6: Quality Review       → code-reviewer 에이전트로 리뷰
Phase 7: Summary              → 결과 정리
```

탐색 → 설계 → 구현 → 리뷰의 순서를 강제한다.

## 언제 무엇을 써야 하는가

### 간단한 버그 수정, 한두 줄 변경

Karpathy만으로 충분하다. Surgical Changes 원칙이 "건드려야 할 것만 건드려라"를 보장하기 때문에 무거운 도구가 필요 없다.

### 기존 코드 수정, 리팩토링

Karpathy + code-simplifier가 적당하다. Karpathy가 과잉 수정을 막고, code-simplifier가 작성 후 자동으로 정리한다.

### 새 기능 개발 (중간 규모)

Karpathy + feature-dev. feature-dev가 탐색 → 설계 → 구현 → 리뷰 순서를 잡아주고, Karpathy가 각 단계에서 코드 품질을 지킨다.

### 대규모 기능 개발, TDD 기반 프로젝트

Superpowers 하나로 커버된다. brainstorming → writing-plans → TDD → subagent-review가 feature-dev + Karpathy의 역할을 모두 한다.

### 정리

| 작업 규모 | 추천 조합 |
|-----------|-----------|
| 한두 줄 수정 | Karpathy |
| 버그 수정 | Karpathy |
| 중간 리팩토링 | Karpathy + code-simplifier |
| 새 기능 (중간) | Karpathy + feature-dev |
| 새 기능 (대규모) | Superpowers |
| TDD 기반 프로젝트 | Superpowers |


## 4개 다 필요한가?

전부 설치해도 충돌하진 않는다. 그렇다고 전부 필요한 것도 아니다.

개인적으로는 Karpathy + Superpowers 조합으로 충분하다고 본다.

첫째, Superpowers가 feature-dev의 상위호환이다. feature-dev의 "탐색 → 설계 → 구현 → 리뷰"를 Superpowers의 brainstorming → writing-plans → TDD → code-review가 더 엄격하게 커버한다. 거기에 TDD 강제, 체계적 디버깅, 합리화 방지까지 붙는다.

둘째, Karpathy와 Superpowers는 상호 보완적이다. Superpowers는 "어떤 프로세스로 일할 것인가"(HOW)를, Karpathy는 "어떤 태도로 코드를 쓸 것인가"(WHAT)를 다룬다. Superpowers가 TDD를 강제해도 테스트 코드 자체가 과잉 엔지니어링될 수 있는데, Karpathy가 이걸 막아준다.

셋째, code-simplifier는 선택사항이다. Superpowers의 code-review + Karpathy의 Simplicity First가 이미 코드 품질을 잡아주기 때문에, 추가 자동 정리가 필요하면 설치해도 좋고 아니면 그만이다.

```
필수: Karpathy + Superpowers
선택: code-simplifier (자동 정리를 원하면)
대체: feature-dev → Superpowers로 대체 가능
```


## 설치 가이드

### Karpathy Guidelines (스킬)

`~/.claude/skills/karpathy-guidelines/` 디렉토리에 `SKILL.md`를 생성하거나, CLAUDE.md에 원칙을 직접 작성한다.

```bash title="GitHub에서 가져오기"
git clone https://github.com/forrestchang/andrej-karpathy-skills.git ~/.claude/skills/karpathy-guidelines
```

### Superpowers (플러그인)

```bash
claude plugin add superpowers
```

세션 시작 시 자동으로 활성화된다. 별도 호출은 필요 없다.

### code-simplifier (플러그인)

```bash
claude plugin add code-simplifier
```

### feature-dev (스킬)

Claude Code 내장 스킬이거나 커뮤니티에서 설치한다. `/feature-dev`로 호출.


## 마치며

4가지 도구는 "더 나은 코드"라는 같은 목적을 향하지만, 각자 다른 레이어에서 동작한다.

```
                    코드 작성 전          코드 작성 중          코드 작성 후
                   ─────────────       ──────────────       ──────────────
Karpathy           가정 명시            최소 코드 원칙        성공 기준 검증
Superpowers        브레인스토밍+플랜     TDD 강제              2단계 리뷰
feature-dev        코드 탐색+설계       구현                  코드 리뷰
code-simplifier    -                   -                    자동 정리
```

- Karpathy: 마인드셋. 항상 켜놓는 기본값
- Superpowers: 방법론. 프로세스를 구조화함
- feature-dev: 워크플로우. 단계를 밟아감
- code-simplifier: 자동화. 끝나면 정리

실무에서는 Karpathy + Superpowers 조합이 가장 무난하다.
